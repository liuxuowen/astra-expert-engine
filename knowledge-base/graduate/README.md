# 研究生申请知识库

按国家/地区分类的研究生申请院校速查、顾问话术与导入规范沉淀。

## 目录结构

```text
graduate/
├── README.md
├── usa/          # 美国研究生
│   └── ...
├── uk/           # 英国研究生
│   └── ...
├── sg/           # 新加坡研究生
│   ├── README.md (本文件)
│   ├── master/   # 硕士项目
│   │   └── nus-master-2026-2027.md
│   └── doctor/  # 博士项目
│       └── nus-doctor-2026-2027.md
├── hk/           # 中国香港研究生
│   └── ...
├── au/           # 澳大利亚研究生
│   └── ...
├── ca/           # 加拿大研究生
│   └── ...
├── eu/           # 欧洲（德国、荷兰、法国等）
│   └── ...
└── other/        # 其他国家/地区
    └── ...
```

## 新加坡研究生目录使用说明
- `sg/master/` 存放硕士项目知识卡
- `sg/doctor/` 存放博士/研究型硕博知识卡
- 每所院校按申请层次（master / doctor）分别建立文件
- 跨层次共用的通用信息（如学校整体认知）可写入同校 master 文件，再用 cross-reference 指向 doctor 文件

## 文件组织原则
- 每个国家/地区对应一个子目录
- 每所院校原则上一个主文件，沉淀该院校的研究生申请核心信息
- 院校文件建议命名为：`{school}-{level}-{season}.md`
  - 例：`nus-graduate-2026-2027.md`
  - 例：`ic-graduate-2026.md`
  - 例：`cuhk-graduate-2026.md`
- 同一院校的新增信息，优先追加到既有主文件，不轻易拆成多个碎片文件

## 单校文件推荐结构
每个院校主文件建议按以下层次组织：

1. **顾问可直接对外讲的咨询话术版**
2. **三段速查版**
   - 申请要求
   - 奖学金/资助
   - 国际生适配
3. **家长沟通版话术**（适用场景：家长为主参与的咨询）
4. **学生沟通版话术**（适用场景：学生本人主导的咨询）
5. **敏感专业/政策提示**（适用场景：涉及 CS、AI、EE 等敏感方向时）
6. **知识库使用建议 / 后续扩展建议**

## 导入体系
后续投喂院校 PDF、截图、网页或文字内容时，统一按以下流程导入：

### Step 1：识别基础归类
先判断以下三件事：
- **国家/地区**：写入对应子目录（如 `sg/`、`hk/`、`uk/`）
- **院校**：并入该院校主文件
- **阶段**：研究生内容统一进入 `knowledge-base/graduate/`

### Step 2：确认申请层次
- 硕士（Master / MS / MA / MBA 等）
- 博士（PhD / Doctor）
- 研究型硕士 vs 授课型硕士（英国、澳洲等体系需区分）
- 专业硕博项目（法学 JD、MBA、医学等）

### Step 3：提炼四层内容
1. **事实层**：学校排名、项目设置、官方要求、申请流程、奖学金规则
2. **判断层**：哪些信息值得顾问重点关注，哪些适合咨询口径复用
3. **话术层**：整理成能直接对家长/学生说的话
4. **边界层**：哪些不能说满，哪些要提醒“以官网为准”

### Step 4：特别关注敏感专业
研究生阶段涉及以下方向时，需要额外标注政策风险提示：
- CS（计算机科学）
- EE（电气工程）
- AI（人工智能）
- 生物医学（Biomedical / Biotechnology）
- 部分涉及国家安全或关键技术的研究方向

### Step 5：写入单校主文件
原则上按以下顺序写入：
- 先更新“顾问可直接对外讲的咨询话术版”
- 再更新“三段速查版”
- 再补“家长沟通版 / 学生沟通版”
- 有敏感专业方向的补充“敏感专业/政策提示”
- 最后补充使用边界、后续待扩展项

## 命名与写作规范
- 用中文写作，保留必要英文项目名称
- 先给结论，再给解释
- 尽量形成可直接复用的话术，而不是单纯摘抄材料
- 奖学金、Funding、TA/RA、Service Obligation 必须区分清楚
- 涉及时间、金额、政策、申请节点时，尽量注明“对应申请季”
- 若内容来自官方 brochure / prospectus，默认可作为高可信信息，但仍建议保留“以官网最新要求为准”的边界表达
